# Architecture Review: Natural Speech Synthesis for Long-Form Audiobooks

## Overview
This document outlines the intended architecture and reasoning for Echomancer v2's long-form text-to-speech (TTS) generation pipeline. The goal is to produce natural, human-sounding audiobooks from PDF text using the F5-TTS model, avoiding mechanical or robotic artifacts at chunk boundaries.

Please review the codebase (specifically `src/lib/generate-audiobook-v2.ts` and `src/lib/gemini-qa.ts`) to verify if the implementation aligns with these principles and if there are any edge cases or flaws that would result in unnatural audio.

## Core Principles & Expected Implementation

### 1. Smart Text Splitting (Zero Overlap)
- **Concept:** Long text must be chunked (e.g., ~1500 characters) to fit within TTS model limits. 
- **Requirement:** Chunks must be split *strictly* at natural linguistic boundaries (paragraphs or terminal punctuation like periods, exclamation marks, question marks).
- **Requirement:** There must be **zero text overlap** between chunks. Overlapping text causes the narrator to repeat the same sentence at the start of a new chunk when the final audio is concatenated.
- **Code to Review:** `splitTextSmart` in `generate-audiobook-v2.ts`.

### 2. Audio Concatenation (Pure Binary, No Crossfade)
- **Concept:** F5-TTS natively generates a natural pause or "breath" (a few hundred milliseconds of silence) at the end of a sentence ending with terminal punctuation.
- **Requirement:** Because the text is chunked at sentence boundaries, the resulting MP3 chunks inherently contain this natural silence at their edges. Therefore, the chunks must be concatenated using **pure binary MP3 concatenation** (pasting the bytes end-to-end).
- **Requirement:** Audio crossfading must **not** be used. Crossfading without text overlap would force the end of Chunk A to play simultaneously with the beginning of Chunk B, clipping spoken words and destroying the natural pause.
- **Requirement:** When binary concatenating, ID3 headers must be stripped from all chunks except the first one to prevent MP3 corruption.
- **Code to Review:** `concatenateSections` and `stripID3Header` in `generate-audiobook-v2.ts`.

### 3. Self-Correcting QA Loop (Gemini 1.5 Flash)
- **Concept:** To prevent compounding errors and bad output, every generated chunk is evaluated by a lightweight LLM (Gemini 1.5 Flash) immediately after generation.
- **Requirement:** If a chunk fails QA (e.g., robotic cadence, wrong emotion, bad pronunciation), the pipeline must catch the failure and retry generating that specific chunk.
- **Requirement:** The QA agent provides actionable feedback (speed multiplier, rewritten text for better phonetic pacing). This feedback must be actively applied to the subsequent retry attempt.
- **Code to Review:** `generateWithRetry` in `generate-audiobook-v2.ts` and `evaluateAudioChunk` in `gemini-qa.ts`.

### 4. Bounded Variations (Seamless Transitions)
- **Concept:** While the QA loop can adjust the speed of a failed chunk to fix pacing, a drastic speed change between Chunk A (e.g., 1.0x) and Chunk B (e.g., 1.5x) will sound like two different narrators, ruining the seamless audiobook experience.
- **Requirement:** Speed adjustments suggested by the QA agent must be strictly clamped/bounded (e.g., between `0.85x` and `1.15x`) to ensure the transition between consecutive chunks remains unnoticeable to the human ear.
- **Code to Review:** The prompt instructions in `evaluateAudioChunk` and the clamping logic in `generateWithRetry`.

## Questions for the Reviewing Agent
1. **Splitting Logic:** Does `splitTextSmart` robustly handle all edge cases for zero-overlap splitting at terminal punctuation? Are there cases where it might split a thought mid-sentence?
2. **Concatenation Logic:** Is the binary concatenation via `Buffer.concat(strippedParts)` mathematically sound for combining MP3s generated by F5-TTS? Does `stripID3Header` accurately remove all non-audio metadata from subsequent chunks?
3. **QA Loop:** Does the exponential backoff and retry mechanism in `generateWithRetry` correctly apply the Gemini text/speed feedback without mutating the original text for *other* chunks in the batch?
4. **General Naturalness:** Based on the current code, is there any remaining architectural reason why the concatenated output would sound mechanical, glitchy, or less than human?
